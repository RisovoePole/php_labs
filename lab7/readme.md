### Шаг 2. Использование готового шаблонизатора (Twig)

#### 2.1. Установка Twig

Создан `composer.json`:

```json
{
    "require": {
        "php": ">=8.0",
        "twig/twig": "^3.0"
    }
}
```

Команда для установки:
```bash
composer install
```

#### 2.2. Twig-шаблоны с наследованием

**templates_twig/base.html.twig** — базовый шаблон:

```twig
<!DOCTYPE html>
<html lang="ru">
<head>
    <title>{% block title %}Рецепты{% endblock %}</title>
</head>
<body>
    {% block content %}{% endblock %}
</body>
</html>
```

**templates_twig/form.html.twig** — наследует base и переопределяет блоки:

```twig
{% extends "base.html.twig" %}

{% block title %}Добавить рецепт - Рецепты{% endblock %}

{% block content %}
<h2>Добавить новый рецепт</h2>
{# форма #}
{% endblock %}
```

Преимущества Twig:
- Более чистый синтаксис.
- Автоматическое экранирование (XSS-защита).
- Встроенные фильтры (upper, lower, date и т.д.).
- Удобное наследование шаблонов через `extends` и `blocks`.

#### 2.3. Инициализация Twig (index-twig.php)

```php
use Twig\Loader\FilesystemLoader;
use Twig\Environment;

$loader = new FilesystemLoader(__DIR__ . '/templates_twig');
$twig = new Environment($loader);

// Рендеринг
echo $twig->render('form.html.twig', [
    'success' => $success,
    'errors' => $errors,
]);
```

### Шаг 3. Создание кастомного Twig-фильтра

Вывод категории рецепта с эмодзи — частая задача. Создан фильтр `recipe_category_icon`:

**src/TwigFilters.php:**

```php
class TwigFilters
{
    /**
     * Фильтр для вывода категории с иконкой.
     * 
     * breakfast → 🌅 Завтрак
     * lunch → ☀️ Обед
     * dinner → 🌙 Ужин
     * dessert → 🍰 Десерт
     */
    public static function recipeCategoryIcon(string $category): string
    {
        $icons = [
            'breakfast' => '🌅 Завтрак',
            'lunch' => '☀️ Обед',
            'dinner' => '🌙 Ужин',
            'dessert' => '🍰 Десерт',
        ];
        return $icons[$category] ?? ucfirst($category);
    }
}
```

**Регистрация фильтра в index-twig.php:**

```php
$twig->addFilter(new \Twig\TwigFilter(
    'recipe_category_icon', 
    [TwigFilters::class, 'recipeCategoryIcon']
));
```

**Использование в шаблоне (list.html.twig):**

```twig
<td>{{ recipe.category | recipe_category_icon }}</td>
```

Вывод: `🌅 Завтрак` вместо просто `breakfast`.

## Сравнение подходов

| Аспект | Нативные PHP-шаблоны | Twig |
|--------|-------------------|------|
| **Синтаксис** | PHP-код, знаком разработчикам | DSL, более понятный для дизайнеров |
| **Безопасность** | Нужно вручную экранировать | Автоматическое экранирование |
| **Производительность** | Быстрее (нет компиляции) | Чуть медленнее (компиляция в PHP) |
| **Наследование** | Через include/require | Встроенное через `extends` и `blocks` |
| **Фильтры** | Нужно писать функции вручную | Встроенные + легко регистрировать свои |
| **Кривая обучения** | Низкая (это PHP) | Средняя (нужно изучить DSL) |

## Контрольные вопросы

### 1. Что такое шаблонизация и зачем она нужна в веб-приложениях?

Шаблонизация — это процесс отделения логики применения (обработка данных, вычисления) от логики представления (как выглядит содержимое на странице).

Зачем она нужна:
- **Разделение ответственности** — логика отдельно, представление отдельно.
- **Переиспользование** — один шаблон может использоваться для разных наборов данных.
- **Удобство** — дизайнеры могут работать с шаблонами, не понимая backend-логику.
- **Безопасность** — централизованное экранирование и обработка данных.
- **Поддержка** — Проще обновлять дизайн и логику независимо.

### 2. Какие преимущества и недостатки нативных PHP-шаблонов в сравнении с шаблонизаторами вроде Twig?

**Преимущества нативных PHP-шаблонов:**
- Простота — это просто PHP, не нужно изучать новый синтаксис.
- Производительность — нет слоя компиляции, прямое выполнение.
- Гибкость — полная мощь PHP доступна в шаблоне.

**Недостатки:**
- Смешивание логики и представления — легко добавить сложную логику в шаблон.
- Безопасность — нужно вручную экранировать каждый вывод.
- Синтаксис — может быть загромождён PHP-кодом, непонятен дизайнерам.

**Преимущества Twig:**
- Безопасность — автоматическое экранирование.
- Чистота — простой DSL без PHP-деталей.
- Наследование — удобное управление общими макетами через `extends` и `blocks`.
- Фильтры и функции — встроенные утилиты.

**Недостатки:**
- Производительность — требует компиляции.
- Кривая обучения — новый синтаксис нужно изучать.
- Зависимость — нужно подключать библиотеку через Composer.

### 3. Как фильтры работают в Twig и как создать свой кастомный фильтр?

Фильтр в Twig — это функция, которая преобразует значение в шаблоне. Синтаксис:

```twig
{{ value | filter_name }}
{{ value | filter_name(arg1, arg2) }}
```

**Встроенные фильтры:**
```twig
{{ "hello" | upper }}          → "HELLO"
{{ 2024-05-15 | date("Y-m-d") }} → "2024-05-15"
```

**Кастомный фильтр создаётся в PHP и регистрируется в Twig:**

```php
// Определение
class TwigFilters {
    public static function myFilter(string $value): string {
        return strtoupper($value);
    }
}

// Регистрация
$twig->addFilter(new \Twig\TwigFilter('my_filter', [TwigFilters::class, 'myFilter']));
```

**Использование в шаблоне:**
```twig
{{ text | my_filter }}
```

В данной работе был ре

ализован фильтр `recipe_category_icon`, который преобразует категорию в строку с эмодзи и названием.

## Ответы на дополнительные вопросы

### 1. В чём отличие нативных PHP-шаблонов от шаблонизатора Twig? Какие преимущества и недостатки у каждого подхода?

Нативные PHP-шаблоны используют обычный PHP-код внутри представления. Twig — это отдельный шаблонизатор с собственным синтаксисом, который компилируется в PHP.

**Нативные PHP-шаблоны:**
- Плюсы: простота старта, высокая гибкость, не нужны внешние зависимости.
- Минусы: легко смешать бизнес-логику и HTML, выше риск пропустить экранирование, сложнее поддерживать большие шаблоны.

**Twig:**
- Плюсы: более чистый и читаемый синтаксис, встроенное экранирование, удобное наследование шаблонов и фильтры.
- Минусы: дополнительная зависимость (Composer), отдельный синтаксис, небольшие накладные расходы на компиляцию.

### 2. Зачем разделять логику и представление в проекте? Какие проблемы могут возникнуть, если смешивать их в одном файле?

Разделение логики и представления делает код проще для поддержки, тестирования и расширения. Логика отвечает за обработку данных, а шаблоны — только за отображение.

Если всё смешано в одном файле, часто возникают проблемы:
- сложнее читать и понимать код;
- трудно вносить изменения без побочных эффектов;
- повышается риск дублирования кода;
- сложнее тестировать бизнес-логику;
- возрастает вероятность ошибок безопасности (например, пропущенное экранирование вывода).

### 3. Что такое наследование шаблонов в Twig? Как работают {% extends %} и {% block %}?

Наследование шаблонов в Twig позволяет вынести общий каркас страницы (header, footer, базовая разметка) в один базовый шаблон и переиспользовать его в дочерних шаблонах.

- `{% extends "base.html.twig" %}` подключает базовый шаблон как основу.
- `{% block content %}...{% endblock %}` определяет именованный участок, который можно переопределить в дочернем шаблоне.

Таким образом, общая структура хранится в одном месте, а в дочерних шаблонах меняется только нужное содержимое блоков.
